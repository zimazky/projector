# Анализ `ZCron.ts` (cron-подобный синтаксис повторов)

## Контекст и назначение

`ZCron.ts` используется для **повторяющихся событий** и решает две задачи:

- **Проверка “срабатывает ли повтор в конкретный день”**: `ZCron.isMatch(scheduleString, startTimestamp, dateTimestamp)`.
- **Поиск первого совпадения на интервале / от даты**: `first`*, `ariseInInterval`.

Использования в проекте:

- В форме события `repeat` валидируется через `ZCron.validate`.
- При рендеринге календаря повторяемые события включаются, если `ZCron.isMatch(...)` возвращает `true`.

Важно: вся логика “по дням” основана на `timestamp` в секундах и шаге `86400`. В библиотеке `DateTime` используется **фиксированный сдвиг часового пояса** (`DateTime.timezone` в часах), **без DST** и без исторических изменений TZ — это влияет на корректность, но делает 1\ \text{день} = 86400 внутренне консистентным.

## Текущая семантика синтаксиса (как реализовано)

Строка расписания ожидается как **три поля**: `days months weekdays`, где поля могут отсутствовать и тогда подставляются `'*'`:

- `days`: 1–31, поддержка `*`, список через `,`, диапазон `a-b`, шаг `a/b`, шаг от начала `*/b`
- `months`: 1–12, те же операторы (кроме спец-режима)
- `weekdays`: 0–6 (0 = Sunday), те же операторы

Дополнительно есть **спец-режим периодичности от старта**:

- если первое поле начинается с `'/'`, например `'/4'`, то расписание означает: “каждые 4 дня от `startTimestamp`”
- при этом **месяцы и дни недели игнорируются** (даже если они указаны)

Логика совпадения в “обычном режиме” — это **AND** между тремя условиями:

`month ∈ months` И `day ∈ days` И `weekday ∈ weekdays`.

## Недостатки и конкретные проблемы (с привязкой к коду)

### 1) Критический баг в валидации чисел

В `testInt` есть ошибка:

- вместо `if (n < min) return false` написано `if (n < min) false`
- из-за этого значения ниже минимума (например, `0` для дней/месяцев) могут проходить `validate(...)`

Практический эффект:

- `ZCron.validate("0 * *")` может вернуть `true`, хотя `0` для day-of-month нелегален
- то же касается `'/0'` (период 0 дней)

### 2) Некорректный разбор по пробелам (ломает пример из комментариев)

И `isMatch`, и `validate` используют `split(' ')` / `split(' ', 3)`:

- это **не эквивалентно “разделить по любому количеству пробельных символов”**
- двойные пробелы создают пустые токены и сдвигают поля

В комментарии к `ZCron.ts` есть пример с двойным пробелом: `'25  2,3 *'`.

- `validate` в текущем виде отклонит его (получится 4 токена: `["25", "", "2,3", "*"]`)
- `isMatch` распарсит поля неверно (месяц станет пустой строкой)

### 3) Off-by-one по дням недели при `*` и шагах

`isMatch` строит `weekdays` так:

- `ZCron.addSequence(..., max=7, from=0)`
- при `'*'` `addSequence` добавляет значения **включая max**, т.е. `0..7`

Такое “лишнее” значение `7` обычно не ломает `includes`, но это:

- нарушение заявленного диапазона 0–6
- источник странностей при шагах (например, `*/7` будет включать 7)

### 4) Валидация и фактическое поведение расходятся по “пустой строке”

`ZCron.validate('')` возвращает `true` (явно разрешено),
но `ZCron.isMatch('')` фактически всегда возвращает `false` (потому что `days` получается пустым).

Это приводит к UX-эффекту “форма пропускает значение, но повторов не будет”.

### 5) “Молчаливое принятие” частично некорректных выражений

`addSequence` и тесты явно допускают строки вроде:

- `'1-5-10'` → трактуется как `'1-5'` (хвост отбрасывается из-за `split('-', 2)`)
- `'1-5-'` → тоже трактуется как `'1-5'`

С точки зрения современных подходов это спорно:

- **пользовательская ошибка не обнаруживается**
- результат “молча другой”, что ухудшает поддержку и отладку

### 6) Производительность: линейный перебор по дням

`ariseInInterval`, `firstInInterval`, `first` идут по дням шагом `86400` и каждый день пересчитывают массивы `days/months/weekdays`.

Проблемы:

- для больших интервалов это O(N) по дням
- пересоздание массивов на каждый вызов `isMatch` — лишняя работа

В текущем проекте это может быть заметно, если:

- много повторяемых событий
- календарь рендерит длинные диапазоны, часто вызывая `isMatch`

### 7) API/дизайн: нет “компиляции” и нет диагностик ошибок

Сейчас:

- `validate()` возвращает только boolean без причины ошибки
- `isMatch()` каждый раз заново “парсит” строку

Современный подход для подобных задач — разделить на:

- **parse/compile** (один раз) → структуру/AST/предикат
- **match** (много раз) → быстрый вызов

## Предложения по улучшению (приоритетно)

### Быстрые и безопасные исправления (минимальный риск, высокая отдача)

- Исправить `testInt`: добавить `return false` для `n < min`.
- В `validate` и `isMatch` заменить разбор полей на разделение по пробельным символам: `trim().split(/\s+/, 3)`.
- В `isMatch` для `weekdays` вызывать `addSequence(..., max=6, from=0)` (или изменить саму `addSequence` на “max inclusive”/“max exclusive” и единообразно использовать).
- Определиться с семантикой пустой строки:
  - либо `validate('')` должно возвращать `false`,
  - либо `isMatch('')` должно трактовать пустую строку как `'* * *'` (т.е. “каждый день”).
- Ужесточить обработку мусорного ввода (как минимум валидацией), чтобы строки типа `'1-5-10'` не считались валидными “частично”.

### Улучшения качества API (без изменения текущей семантики совпадения)

- Добавить `ZCron.parse(scheduleString)` → возвращает структуру:
  - режим: `'relative' | 'absolute'`
  - наборы `days/months/weekdays` (лучше в виде `Set<number>` или битмасок для 0..6)
  - нормализованные значения (отсортированные, без дубликатов)
- Добавить `ZCron.match(parsed, date)` и/или внутренний кэш `Map<string, ParsedSchedule>`:
  - при рендеринге календаря это уберёт повторный парсинг одной и той же строки
- Сделать валидацию информативной:
  - `validateDetailed()` → `{ ok: boolean, error?: { field, token, message } }`
  - это значительно упрощает UX в форме и поддержку

### Оптимизация вычислений “следующего срабатывания”

Если появится необходимость (много событий / длинные интервалы), вместо перебора:

- реализовать `nextAfter(schedule, startTimestamp, afterTimestamp)`:
  - “перепрыгивание” по месяцам/дням/неделям на ближайший кандидат
  - для относительного режима `'/d'` это вообще O(1): вычислить k = \lceil\frac{\Delta}{d}\rceil и вернуть `start + k*d*86400`

## Тестирование (что стоит добавить)

Сейчас тесты покрывают только `addSequence`. Минимальный набор тестов, который быстро поймает реальные баги:

- `validate`:
  - `0 * `* должно быть `false`
  - `25  2,3 *` (двойные пробелы) должно корректно парситься
  - `'/0'` должно быть `false`
- `isMatch`:
  - кейсы `'/d'` (совпадает в день старта, не совпадает до старта, корректно считает разницу)
  - `weekdays='*'` не должен включать 7 (проверка через match на “несуществующий” weekday невозможна напрямую — лучше проверять именно разбор/компиляцию)
- интеграционный тест на “валидно в форме → реально срабатывает в календаре”

## Итог

`ZCron.ts` — полезная и компактная библиотека под текущие нужды (“повторы по дням”), но в текущем виде имеет **несколько реальных дефектов** (ошибка в `testInt`, парсинг пробелов, off-by-one по weekday) и архитектурные ограничения (нет компиляции/кэша, нет диагностик ошибок, линейные алгоритмы поиска).

Наиболее рациональный путь модернизации:

- быстро исправить баги и парсинг,
- затем добавить слой `parse/compile + cache`,
- после этого (если потребуется) оптимизировать `nextAfter`/поиск в интервалах и расширять синтаксис, не ломая текущую семантику.
