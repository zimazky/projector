### Реализация Google-аутентификации

- **Сервис `GAPI`** (`src/7-shared/services/gapi.ts`):
  - Инкапсулирует низкоуровневую работу с Google API (GIS + gapi client).
  - Инициализирует `google.accounts.oauth2.TokenClient` и `gapi.client`.
  - Управляет логином/логаутом и проверкой токена:
    - `GAPI.logIn(prompt?: string)` — запускает запрос access token.
    - `GAPI.logOut()` — отзывает токен и очищает его в `gapi.client`.
    - `GAPI.isLoggedIn()` — проверяет наличие токена в `gapi.client`.
  - Текущие scope:
    - `https://www.googleapis.com/auth/drive.file`
    - `https://www.googleapis.com/auth/drive.readonly`
    - `https://www.googleapis.com/auth/drive.appfolder`
  - Эти scope предназначены для работы с Google Drive и **не включают** доступ к профилю пользователя (`userinfo.profile` / People API).

- **Сервис `GoogleApiService`** (`src/7-shared/services/GoogleApiService.ts`):
  - MobX-сервис-обёртка над `GAPI`.
  - Отвечает за:
    - Инициализацию (`initGapi`), включая обработчики:
      - `onSuccess` — установка `isGoogleLoggedIn = GAPI.isLoggedIn()`.
      - `onSignIn` — обновление `isGoogleLoggedIn` в реактивном состоянии.
      - `onExpiredToken` — сброс флага логина.
    - Обновление observable поля `isGoogleLoggedIn: boolean`.
    - Вызовы `logIn`, `logOut`, `isLoggedIn`.
    - Высокоуровневые операции с Google Drive (чтение/запись/удаление файлов и папок).
  - Сервис **не хранит данных профиля пользователя** (имя, email, URL аватара).

- **MainStore и инициализация** (`src/1-app/Stores/MainStore.ts`):
  - Содержит приватное поле `googleApiService: GoogleApiService`.
  - В методе `init()`:
    - Вызывает `this.googleApiService.initGapi()`.
    - Ждёт `waitForGapiReady()` и затем восстанавливает последний открытый документ.
  - Таким образом, Google API инициализируется централизованно на уровне приложения.

- **StoreContext** (`src/1-app/Providers/StoreContext.ts`):
  - Интерфейс `IRootStore` включает:
    - `googleApiService: GoogleApiService;`
    - Другие сторы и сервисы.
  - Экспортирует `StoreContext`, через который `Header` уже получает сторы (`calendarStore`, `uiStore`, `documentSessionStore`) и **может получить** `googleApiService` без нарушения FSD.

### Вывод по текущему состоянию

- На уровне инфраструктуры уже есть:
  - Централизованная инициализация Google API.
  - Реактивный флаг `isGoogleLoggedIn` в `GoogleApiService`.
  - Доступ к `googleApiService` из `Header` через `StoreContext`.
- В проекте **нет**:
  - Сущности пользователя (`User`) или отдельного стора с пользовательским профилем.
  - API для получения имени/аватарки пользователя из Google (нет People API / `userinfo`).
  - Визуального компонента для отображения аватара.

Это означает, что:
- Минимальная реализация «иконка пользователя при логине» возможна уже сейчас на основе флага `isGoogleLoggedIn` (с плейсхолдер-аватаром).
- Для показа **реального** Google-аватара потребуется дополнительная интеграция с API профиля и расширение scope, что влияет на настройки проекта в Google Cloud Console.

## Варианты получения данных пользователя и аватара

### Вариант A: Только факт логина (MVP)

Описание:
- Использовать уже имеющийся `isGoogleLoggedIn` из `GoogleApiService`.
- В `Header`:
  - Если пользователь залогинен (`isGoogleLoggedIn === true`) — показывать иконку пользователя (плейсхолдер: инициалы, силуэт, иконка).
  - Если не залогинен — либо ничего не показывать, либо оставить текущее поведение без изменений.

Плюсы:
- Не требует изменения скопов и интеграции дополнительных API.
- Минимальный риск и объём изменений.
- Соответствует текущей архитектуре (всё есть в коде).

Минусы:
- Иконка будет «обезличенной» (без реального фото и имени).

### Вариант B: Реальный профиль пользователя (Google avatar)

Описание:
- После получения access token от GIS использовать дополнительный запрос для получения профиля:
  - Через People API (`people.get`).
  - Или через `https://www.googleapis.com/oauth2/v3/userinfo`.
- Сохранить в состоянии:
  - Имя (`displayName`).
  - URL аватара (`photoUrl` / `picture`).

Требуемые изменения:
1. **Расширить scope авторизации**:
   - Добавить `https://www.googleapis.com/auth/userinfo.profile` или соответствующие scope People API.
   - Обновить конфигурацию в Google Cloud Console (если необходимо).
2. **Расширить `GAPI`**:
   - Добавить метод, который с текущим токеном сделает HTTP-запрос к API профиля (через `fetch` или `gapi.client`).
3. **Расширить `GoogleApiService`**:
   - Добавить observables для `userName`, `userAvatarUrl`.
   - При успешном логине запрашивать и сохранять профиль.
4. **Использовать новые поля в UI**:
   - Визуальный компонент `UserAvatar` будет принимать уже не только флаг логина, но и `avatarUrl`/`userName`.

Плюсы:
- Отображение реального аватара и имени пользователя.

Минусы:
- Более сложная интеграция, зависимость от People API/`userinfo`.
- Требуется изменение scope и, возможно, согласование политики безопасности.

### Рекомендация по этапности

- **Этап 1 (MVP)**: Реализовать отображение плейсхолдер-аватара при `isGoogleLoggedIn === true` без расширения scope.
- **Этап 2 (расширение)**: При необходимости доработать интеграцию для получения реального профиля и заменить источник данных `UserAvatar` на реальные значения.

Ниже план будет описан в первую очередь для MVP (вариант A) с указанием точек расширения под вариант B.

## Тестовый план

### Юнит-/компонентные проверки

- **UserAvatar**:
  - Рендерит `null`, если `isLoggedIn === false`.
  - Рендерит плейсхолдер, если `isLoggedIn === true` и `avatarUrl` не передан.
  - Рендерит `<img>` с корректным `src`, если `avatarUrl` задан.
  - Вызывает `onClick`, если передан обработчик и пользователь кликнул по аватару.

- **Header**:
  - При `googleApiService.isGoogleLoggedIn === false` вёрстка хедера не содержит `UserAvatar`.
  - При `googleApiService.isGoogleLoggedIn === true` `UserAvatar` присутствует и располагается в нужной части хедера.

### Интеграционные/ручные проверки

1. **Логин в Google**:
   - Запустить приложение, убедиться, что GAPI инициализируется.
   - Вызвать сценарий логина (через существующий функционал).
   - Проверить, что после логина:
     - `googleApiService.isGoogleLoggedIn === true`.
     - В Header отображается иконка пользователя.
2. **Логаут** (если реализован в рамках задачи):
   - Кликнуть по аватару/пункту меню «Выйти».
   - Убедиться, что:
     - Вызвался `googleApiService.logOut()`.
     - Флаг `isGoogleLoggedIn` сброшен.
     - Иконка исчезла из Header.
3. **Обновление страницы**:
   - После успешного логина обновить страницу.
   - Проверить, что после повторной инициализации GAPI:
     - `isGoogleLoggedIn` корректно отражает состояние.
     - Иконка в Header появляется/не появляется в соответствии со статусом.
