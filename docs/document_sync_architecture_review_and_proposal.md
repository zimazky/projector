# Анализ вариантов синхронизации документа и предложение по целевой архитектуре

**Дата:** 14 апреля 2026 г.  
**Основание:** анализ документов `document_sync_and_state_indication_analysis.md`, `document_sync_state_machine_proposal.md` и текущей кодовой базы

## 1. Краткий вывод

Оба существующих документа правильно диагностируют симптомы, но решают задачу с разной глубиной:

- `document_sync_and_state_indication_analysis.md` хорошо фиксирует пользовательские проблемы и даёт безопасные точечные улучшения.
- `document_sync_state_machine_proposal.md` сильнее на уровне модели состояния и лучше формулирует корневую причину.

При этом ни один из вариантов не закрывает архитектуру полностью:

- первый документ слишком долго оставляет систему на трёх пересекающихся флагах;
- второй документ слишком жёстко сводит всё к одному FSM-состоянию и смешивает долговременное состояние данных с временными операциями.

Оптимальный путь для этого проекта: не косметический патч флагов и не полный big-bang рефакторинг в один enum, а **единая доменная модель синхронизации с derive-представлением для UI, отдельным состоянием операции и централизованными командами записи**.

## 2. Что реально происходит в коде сейчас

### 2.1 Состояние документа описывается пересекающимися полями

В `DocumentTabsStore.types.ts` документ хранит:

- `isDirty`
- `hasUnsyncedChanges`
- `syncStatus`
- `isLoading`
- `isSaving`
- `error`
- временные метки

Проблема не в количестве полей как таковом, а в том, что они описывают не независимые оси состояния, а одну и ту же предметную область разными способами.

### 2.2 При редактировании теряется знание о более новой версии в Drive

В [src/6-entities/Document/model/DocumentTabsStore.ts](/d:/@projects/projector/src/6-entities/Document/model/DocumentTabsStore.ts:294) при изменении данных:

- `isDirty = true`
- если было `synced`, статус становится `needs-sync`
- если было `update-available`, статус тоже становится `needs-sync`

Это означает, что состояние "на сервере есть новая версия" превращается в обычное "у меня локальные изменения", хотя это уже другая ситуация: потенциальная дивергенция двух версий.

### 2.3 Синхронизация может перезаписать локальные данные без явного решения пользователя

В [src/6-entities/Document/model/DocumentTabsStore.ts](/d:/@projects/projector/src/6-entities/Document/model/DocumentTabsStore.ts:382) логика такая:

1. берутся метаданные файла;
2. сравнивается `remoteModifiedAt > lastSavedAt`;
3. если изменений не найдено, выполняется загрузка контента с Drive;
4. локальные данные заменяются.

Это опасно по двум причинам:

- сравнение завязано только на timestamp;
- ветка "изменений нет" всё равно делает destructive refresh локального состояния.

### 2.4 `hasUnsyncedChanges` фактически не живёт своей жизнью

В [src/6-entities/Document/model/DocumentTabsStore.ts](/d:/@projects/projector/src/6-entities/Document/model/DocumentTabsStore.ts:548) при сохранении snapshot в `localStorage` записывается:

- `hasUnsyncedChanges: session.state.isDirty`

То есть флаг не отражает независимую историю между сессиями, а просто копирует текущее `isDirty`.

### 2.5 Есть второй путь записи, который обходит правила `DocumentTabsStore`

В [src/4-widgets/SaveToDrive/model/SaveToDriveStore.ts](/d:/@projects/projector/src/4-widgets/SaveToDrive/model/SaveToDriveStore.ts:88) после успешного `saveFile()` происходит прямая мутация:

- `activeDoc.ref = ...`
- `activeDoc.state.isDirty = false`
- `activeDoc.state.syncStatus = 'synced'`

Это архитектурно важнее, чем кажется. Даже идеальная модель состояния не поможет, если запись возможна в обход единого доменного API.

### 2.6 UI уже сейчас показывает разные правды в разных местах

- В табах [DocumentTabs.tsx](/d:/@projects/projector/src/7-shared/ui/DocumentTabs/DocumentTabs.tsx:72) отображается `isDirty || hasUnsyncedChanges`.
- В заголовке [Header.tsx](/d:/@projects/projector/src/4-widgets/Header/Header.tsx:21) отображается только `isDirty`.
- В `CalendarIconBar` логика действий и предупреждений завязана на свои условия.

Итог: у системы нет одного UI-контракта "что считать локальными изменениями", "что считать конфликтом" и "когда спрашивать пользователя".

## 3. Оценка документа `document_sync_and_state_indication_analysis.md`

### 3.1 Сильные стороны

- Хорошо описаны пользовательские сценарии и визуальная сторона проблемы.
- Правильно выявлены опасные точки:
  - подмена `activeDocumentId` в `syncAllDocumentsWithDrive()`;
  - потеря `update-available` при редактировании;
  - бесполезность `hasUnsyncedChanges`;
  - отсутствие защиты от гонок.
- Подход эволюционный, а значит реалистичен для проекта с уже существующим UI и тестами.
- Есть здравая идея разделить `check` и `action`.

### 3.2 Слабые стороны

- Документ лечит симптомы, но не устраняет главный источник сложности: тройную модель `isDirty` / `hasUnsyncedChanges` / `syncStatus`.
- Визуальная дифференциация предлагается раньше, чем формализована доменная модель. Это повышает риск "красивого UI поверх хрупкой логики".
- `hasUnsyncedChanges` сохраняется как отдельный смысловой флаг, хотя текущая модель уже показывает, что этот флаг трудно удержать консистентным.
- В ряде мест документ продолжает мыслить через `syncStatus`, который уже перегружен несколькими смыслами: и режим связи, и конфликт, и требование действия.

### 3.3 Итоговая оценка

Это **хороший промежуточный документ**, если цель быстро снизить риск потери данных и немного улучшить UX.  
Это **не лучшая целевая архитектура**, если мы хотим, чтобы система оставалась понятной после добавления новых сценариев.

## 4. Оценка документа `document_sync_state_machine_proposal.md`

### 4.1 Сильные стороны

- Верно определена корневая проблема: текущая система допускает противоречивые комбинации состояний.
- Очень сильная идея: делать некорректные состояния невыразимыми.
- Хорошо замечено, что нельзя сводить конфликт к сравнению только по времени.
- Правильно поднимается тема централизации переходов и удаления прямых мутаций вне `DocumentTabsStore`.

### 4.2 Слабые стороны

- Предложенный FSM слишком монолитен: в один union сведены и долговременное состояние синхронизации, и временная операция (`syncing`), и ошибка.
- Это создаёт риск комбинаторного роста и усложняет практическую разработку UI.
- Состояния `error` и `syncing` в большинстве современных клиентских архитектур лучше держать отдельно от базового статуса согласованности данных.
- Полная замена текущей модели на новый union за один проход выглядит слишком дорогой и рискованной для существующего кода.
- Предложение с fingerprint в правильную сторону, но в документе недооценена стоимость полного пересчёта/сериализации и не разделены:
  - локальный отпечаток текущих данных;
  - отпечаток базовой синхронизированной версии;
  - наблюдённая удалённая ревизия.

### 4.3 Итоговая оценка

Это **более сильный архитектурный документ**, чем первый.  
Но в текущем виде он тянет систему в сторону слишком жёсткой модели, где transient-state и persistent-state смешаны в одном типе.

## 5. Что я предлагаю как целевую модель

### 5.1 Принцип

Нужно моделировать не "иконки и флаги", а **три независимые оси**:

1. **Локальная версия данных**  
   Что сейчас редактирует пользователь.
2. **Базовая синхронизированная версия**  
   От какой версии началось текущее локальное редактирование.
3. **Состояние текущей операции**  
   Идёт проверка, сохранение, загрузка, ошибка или ничего не идёт.

Современный практический подход здесь не "один огромный enum", а **нормализованная доменная модель + вычисляемые селекторы для UI**.

### 5.2 Предлагаемая структура состояния

```ts
type SyncOperation = 'idle' | 'checking' | 'saving' | 'pulling'

type SyncErrorState = {
    code: string
    message: string
    at: number
} | null

type DocumentSyncSnapshot = {
    localFingerprint: string | null
    baseFingerprint: string | null
    remoteFingerprint: string | null
    baseRevisionId: string | null
    remoteRevisionId: string | null
    lastSyncAt: number | null
    origin: 'new-local' | 'drive' | 'restored-local'
}

type DocumentSession = {
    id: DocumentId
    ref: DocumentRef | null
    data: DocumentData
    sync: DocumentSyncSnapshot
    operation: SyncOperation
    error: SyncErrorState
    operationToken: number | null
    createdAt: number
    lastAccessedAt: number
}
```

### 5.3 Что вычисляется, а не хранится флагами

Из этой структуры через селекторы вычисляются:

- `isDirty`: `localFingerprint !== baseFingerprint`
- `hasRemoteAhead`: `remoteRevisionId !== null && remoteRevisionId !== baseRevisionId`
- `isDiverged`: `isDirty && hasRemoteAhead`
- `uiSyncStatus`: `'new' | 'dirty' | 'synced' | 'remote-ahead' | 'conflict' | 'error' | 'syncing'`
- `shouldWarnOnClose`
- `canSave`
- `canPullRemote`

Это важный сдвиг: UI больше не зависит от внутренних флагов напрямую, а зависит от единого набора derive-правил.

## 6. Почему это лучше, чем оба исходных варианта

### 6.1 Лучше, чем текущая флаговая модель

- Нет трёх источников истины для одного смысла.
- Нет необходимости вручную синхронизировать `isDirty` и `hasUnsyncedChanges`.
- Становится понятно, что такое "локально изменён", "сервер новее" и "обе стороны изменены".

### 6.2 Лучше, чем полный FSM в одном union

- Долговременная согласованность данных и временная операция разделены.
- UI проще: статус можно вывести селектором, а не раздувать union множеством подтипов.
- Ошибка становится побочным состоянием операции, а не альтернативной формой жизненного цикла документа.
- Поэтапная миграция проще: можно оставить наружу старые геттеры совместимости.

## 7. Целевые доменные статусы для UI

Показывать в интерфейсе нужно не внутренние поля, а производный `uiSyncStatus`:

| `uiSyncStatus` | Смысл | Условие |
|---|---|---|
| `new` | Новый локальный документ, ещё нигде не сохранён | `origin = new-local` и нет `fileId` |
| `dirty` | Есть локальные изменения относительно базовой версии | `isDirty && !hasRemoteAhead` |
| `synced` | Локальная версия совпадает с базовой и удалённой | `!isDirty && !hasRemoteAhead` |
| `remote-ahead` | На Drive есть более новая версия, локально изменений нет | `!isDirty && hasRemoteAhead` |
| `conflict` | Изменены и локальная, и удалённая стороны | `isDirty && hasRemoteAhead` |
| `syncing` | Идёт операция | `operation !== idle` |
| `error` | Последняя операция завершилась ошибкой | `error !== null` и `operation === idle` |

Именно этот статус должны использовать:

- `DocumentTabs`
- `Header`
- `CalendarIconBar`
- диалог закрытия документа
- конфликтный диалог

## 8. Правила переходов

### 8.1 После открытия из Drive

- загрузить контент;
- вычислить `localFingerprint`;
- записать его же в `baseFingerprint`;
- сохранить `baseRevisionId` из метаданных;
- `operation = idle`, `error = null`.

### 8.2 После локального редактирования

- обновить `session.data`;
- пересчитать `localFingerprint`;
- не трогать `baseFingerprint`;
- не трогать `remoteRevisionId`, если мы уже знаем, что сервер ушёл вперёд.

Ключевой эффект: если документ был `remote-ahead`, после локального редактирования он автоматически становится `conflict`, а не `needs-sync`.

### 8.3 После проверки Drive

Нужен отдельный метод уровня домена:

```ts
checkRemote(documentId): Promise<RemoteCheckResult>
```

Он:

- читает только удалённые метаданные;
- обновляет `remoteRevisionId` и, если доступно, `remoteFingerprint`;
- не подменяет локальные данные;
- не меняет `baseFingerprint`.

### 8.4 После успешного сохранения в Drive

- взять актуальный снимок локальных данных;
- выполнить сохранение;
- после ответа сервера обновить:
  - `baseFingerprint = localFingerprint`
  - `baseRevisionId = remoteRevisionId = revisionFromServer`
  - `lastSyncAt = now`
  - `error = null`

### 8.5 После явной загрузки версии из Drive

- загрузить удалённый контент только по команде пользователя;
- заменить локальные данные;
- пересчитать fingerprint;
- обновить `base*` и `remote*` как новую точку синхронизации.

## 9. Обязательные архитектурные ограничения

### 9.1 Единственная точка записи состояния документа

Любое успешное сохранение должно завершаться через `DocumentTabsStore`, а не через прямую мутацию снаружи.

Нужны методы уровня домена:

- `markDocumentSavedToDrive(documentId, fileMetadata, fingerprint, revisionId)`
- `markRemoteChecked(documentId, remoteInfo)`
- `applyRemoteVersion(documentId, remoteContent, remoteInfo)`
- `markOperationFailed(documentId, error)`

`SaveToDriveStore` должен только запускать сценарий, а не менять `activeDoc.state` напрямую.

### 9.2 `syncAllDocumentsWithDrive()` не должен менять active tab

Синхронизация должна работать по `documentId`, а не через временную подмену `activeDocumentId`.

### 9.3 Нельзя автоматически делать pull после check

Проверка удалённого состояния и применение удалённой версии должны быть разными командами.

### 9.4 Нужна защита от устаревших async-результатов

`operationToken` или аналогичный request id обязателен для:

- `save`
- `checkRemote`
- `pullRemote`

Устаревший ответ не должен менять текущее состояние документа.

## 10. Практический план внедрения

### Этап 1. Немедленные исправления без смены публичного API

1. Запретить прямые мутации из `SaveToDriveStore`; перевести их на методы `DocumentTabsStore`.
2. Убрать автоматический `downloadFileContent()` из ветки "изменений нет" в `syncActiveDocumentWithDrive()`.
3. Перестать сбрасывать `update-available` в обычный `needs-sync` при редактировании.
4. Убрать подмену `activeDocumentId` из `syncAllDocumentsWithDrive()`.
5. Исправить запись `hasUnsyncedChanges` в snapshot, чтобы она хотя бы не копировала `isDirty`.

Этот этап снижает риск потери данных даже до полной смены модели.

### Этап 2. Введение новой внутренней модели

1. Добавить в `DocumentSession` поля `sync`, `operation`, `error`, `operationToken`.
2. Реализовать селекторы:
   - `getIsDirty(session)`
   - `getUiSyncStatus(session)`
   - `getShouldWarnOnClose(session)`
   - `getHasConflict(session)`
3. Перевести `DocumentTabs`, `Header`, `CalendarIconBar` на селекторы.
4. Оставить старые поля только как адаптер на переходный период.

### Этап 3. Разделение команд

Нужны отдельные методы:

- `checkDocumentRemoteState(documentId)`
- `saveDocumentToDrive(documentId)`
- `pullDocumentFromDrive(documentId)`
- `resolveConflictByKeepingLocal(documentId)`
- `resolveConflictByApplyingRemote(documentId)`

`syncActiveDocumentWithDrive()` как "магическая всё-в-одном команда" лучше удалить после миграции.

### Этап 4. Очистка legacy

1. Удалить `hasUnsyncedChanges`.
2. Удалить старый `syncStatus`.
3. Удалить прямые чтения `isDirty` в UI.
4. Перевести snapshot в `localStorage` на новую схему.

## 11. Когда и как приложение должно проверять Drive

### 11.1 Упрощение модели

Идея не отслеживать отдельный `offline`-статус действительно здравая и для этого проекта, на мой взгляд, полезная.

Если сети нет:

- пользователь всё равно не сможет войти в Google;
- `check/save/pull` просто завершатся ошибкой доступа;
- для пользовательского сценария это почти всегда выглядит одинаково: "Drive сейчас недоступен".

Поэтому доменную модель можно упростить и **не хранить сеть как самостоятельный режим приложения**. Для синхронизации нам важнее другое:

- есть ли сейчас подтверждённая возможность работать с Drive;
- нужно ли перепроверить удалённое состояние документа;
- чем завершилась последняя remote-операция.

### 11.2 Базовый принцип проверки

Проверка Drive должна быть:

- **событийной**, а не по таймеру "каждые N секунд";
- **ленивой**, то есть запускаться тогда, когда это реально даёт пользовательскую ценность;
- **безопасной**, то есть не менять локальные данные автоматически;
- **дедуплицированной**, чтобы один и тот же документ не уходил в параллельные проверки.

Иными словами, приложению нужен не "автосинк", а **авто-check + явный user-driven action**.

### 11.3 Когда обязательно запускать проверку Drive

#### 1. При переходе из `drive-unavailable` в `drive-available`

Это главный триггер. Как только приложение снова может работать с Drive, нужно:

- пройтись по документам, связанным с Drive;
- поставить их в очередь фоновой проверки;
- проверить сначала активный документ, затем остальные с ограничением параллелизма.

Почему это важно:

- пока приложение не могло работать с Drive, удалённая версия могла измениться;
- локальные документы, восстановленные из `localStorage`, могли устареть;
- пользователь ожидает, что после восстановления доступа статусы станут актуальными.

На практике этот переход чаще всего происходит, когда:

- пользователь только что выполнил вход в Google;
- токен был просрочен, а затем восстановлен;
- предыдущая remote-операция упала по сетевой причине, а следующая успешная попытка показала, что доступ к Drive снова есть.

#### 2. При открытии документа из Drive

Это обязательная проверка по факту самого открытия:

- получаем метаданные;
- загружаем содержимое;
- фиксируем базовую ревизию и fingerprint.

Этот сценарий не должен считаться "фоновым check", потому что здесь пользователь явно запросил работу с удалённым документом.

#### 3. При активации вкладки, если состояние удалённой версии считается устаревшим

Не нужно проверять Drive при каждом переключении вкладки. Это будет шумно и дорого.

Лучше ввести TTL для удалённой проверки, например:

- если документ ни разу не проверялся с момента последнего восстановления доступа к Drive;
- или `lastRemoteCheckAt` старше заданного окна;
- или документ был восстановлен из `localStorage` и ещё не проходил remote-check после старта приложения.

Тогда при активации вкладки:

- если документ "свежий", проверку не делать;
- если документ "протух", ставить только этот документ в `check`.

Это даёт хороший баланс между актуальностью и лишними запросами.

#### 4. Перед операциями, которые могут привести к потере удалённых изменений

Перед:

- `saveDocumentToDrive(documentId)`,
- `resolveConflictByKeepingLocal(documentId)`,
- массовым сохранением,

нужно убедиться, что у нас есть достаточно свежая информация о состоянии Drive.

Практически это означает:

- если удалённая проверка устарела, сначала делаем `checkDocumentRemoteState(documentId)`;
- только потом разрешаем `save`.

Так мы не сохраняем локальную версию поверх неожиданно изменившейся удалённой версии.

### 11.4 Когда проверку делать не нужно

Проверку не стоит запускать:

- на каждый локальный edit;
- на каждый `autosave` в `localStorage`;
- на каждое переключение таба без TTL/dirty-условия;
- сразу после `save`, если сервер уже вернул новую ревизию;
- когда Drive явно недоступен и пользователь не инициировал действие, требующее повторной попытки.

Особенно важно не превращать редактирование в постоянный фоновой опрос Drive.

### 11.5 Какой вариант реализации я считаю лучшим

Я бы предложил ввести отдельный сервис уровня приложения, например `DocumentConnectivityCoordinator` или `DocumentSyncScheduler`.

Его ответственность:

- отслеживать прикладную доступность Drive: `drive-unavailable | drive-available`;
- слушать:
  - изменения `googleApiService.isGoogleLoggedIn`;
  - успешные и неуспешные remote-операции;
  - активацию документа;
- планировать `checkDocumentRemoteState(documentId)`;
- не выполнять `save/pull`, только планировать безопасные проверки.

### 11.6 Предлагаемая модель доступности Drive

```ts
type AppDriveAvailability =
    | 'drive-unavailable'
    | 'drive-available'
```

Базовое правило:

- `drive-available`: есть валидная авторизация, и remote-операции в данный момент не находятся в состоянии подтверждённой недоступности;
- `drive-unavailable`: все остальные случаи.

Важно: это специально грубая модель. Она проще, чем отдельное хранение "есть сеть", "есть логин", "истёк токен", "временная ошибка API", и для клиентского поведения здесь этого достаточно.

### 11.7 Предлагаемый алгоритм планирования проверки

```ts
onDriveAvailabilityChanged(prev, next) {
    if (next !== 'drive-available') return

    scheduleCheckForActiveDocument('drive-became-available')
    scheduleBackgroundCheckForRestorableDriveDocuments('drive-became-available')
}

onActiveDocumentChanged(documentId) {
    if (appDriveAvailability !== 'drive-available') return
    if (!shouldCheckRemote(documentId)) return

    scheduleCheck(documentId, 'document-activated')
}

beforeSaveDocument(documentId) {
    if (appDriveAvailability !== 'drive-available') return 'block-or-save-locally'
    if (!isRemoteCheckFresh(documentId)) {
        await checkDocumentRemoteState(documentId)
    }
}
```

### 11.8 Правило `shouldCheckRemote(documentId)`

Для документа, связанного с Drive, `shouldCheckRemote(documentId) === true`, если выполняется хотя бы одно условие:

- документ восстановлен из `localStorage` и ещё не был проверен после старта приложения;
- приложение только что вошло в режим `drive-available`;
- `lastRemoteCheckAt` отсутствует;
- `lastRemoteCheckAt` старше TTL;
- предыдущая проверка завершилась ошибкой доступа к Drive, и сейчас доступность восстановлена;
- перед сохранением нужна актуализация remote-состояния.

### 11.9 Какой TTL я бы предложил

Для этого проекта разумно начать с простого правила:

- активный документ: TTL 30-60 секунд;
- неактивные открытые Drive-документы: проверка только при входе в `drive-available` и при явной активации.

Почему так:

- активный документ чаще всего является предметом текущего редактирования;
- фоновые проверки всех вкладок при каждом чихе будут раздражать и загружать API;
- для неактивных документов важнее "проверить при возвращении к ним", чем держать их идеально свежими каждую секунду.

### 11.10 UX-правило для смены доступности Drive

При смене доступности Drive интерфейс должен менять не данные, а только доступность действий и статус:

- при переходе в `drive-unavailable`:
  - не трогаем локальные данные;
  - переводим UI в режим недоступности remote-операций;
  - можем показывать "статус Drive может быть устаревшим";
- при переходе в `drive-available`:
  - запускаем `check`, но не открываем конфликтный диалог автоматически для всех документов подряд;
  - активному документу можно показать обновлённый статус сразу;
  - для фоновых документов лучше ограничиться обновлением индикаторов.

### 11.11 Что нужно добавить в состояние документа

Чтобы такая схема работала, в `DocumentSession.sync` стоит хранить ещё несколько полей:

```ts
type DocumentSyncSnapshot = {
    localFingerprint: string | null
    baseFingerprint: string | null
    remoteFingerprint: string | null
    baseRevisionId: string | null
    remoteRevisionId: string | null
    lastSyncAt: number | null
    lastRemoteCheckAt: number | null
    needsRemoteCheck: boolean
    origin: 'new-local' | 'drive' | 'restored-local'
}
```

Смысл:

- `lastRemoteCheckAt` нужен для TTL;
- `needsRemoteCheck` удобен как явный флаг после восстановления из `localStorage`, ошибок доступа к Drive и смены доступности;
- `origin` помогает различать "новый локальный документ" и "восстановленный Drive-документ".

### 11.12 Конкретная рекомендация

Я бы внедрял проверку Drive так:

1. Добавить `AppDriveAvailability` и простой coordinator, который слушает логин и результаты remote-операций.
2. Разделить `checkDocumentRemoteState()` и `pullDocumentFromDrive()`.
3. Ввести `lastRemoteCheckAt` и `needsRemoteCheck`.
4. Делать массовый `check` только при переходе в `drive-available`.
5. При активации вкладки проверять Drive только по TTL/флагу устаревания.
6. Перед `save` обязательно гарантировать свежий `check`.

Это даст предсказуемое поведение:

- отсутствие отдельного offline-режима не усложняет модель;
- восстановление доступа к Drive актуализирует статусы;
- приложение не спамит Drive лишними запросами;
- конфликт определяется в нужный момент, а не случайно и не слишком поздно.

## 12. Что я бы не рекомендовал делать

- Не вводить новую богатую визуальную схему раньше новой доменной модели.
- Не делать полный переход на один большой `DocSyncState` без поэтапной миграции.
- Не полагаться только на `modifiedTime`.
- Не использовать `activeDocumentId` как технический рычаг для пакетных операций.
- Не оставлять два независимых пути изменения `DocumentSession`.

## 13. Минимальный целевой API для UI

Чтобы UI перестал знать о внутренней кухне, достаточно дать ему такой контракт:

```ts
type DocumentUiState = {
    status: 'new' | 'dirty' | 'synced' | 'remote-ahead' | 'conflict' | 'syncing' | 'error'
    showModifiedMark: boolean
    showSyncIcon: boolean
    titleSuffix: string
    tooltip: string
    canSave: boolean
    canSync: boolean
}
```

Тогда:

- табы перестанут комбинировать `isDirty` и `hasUnsyncedChanges` вручную;
- заголовок и toolbar будут согласованы;
- конфликтный диалог сможет открываться по нормализованному признаку `status === 'conflict'`.

## 14. Итоговая рекомендация

Если выбирать между двумя существующими документами как базой:

- **как краткосрочную основу** я бы взял `document_sync_and_state_indication_analysis.md`, потому что он проще для безопасных первых фиксов;
- **как направление целевой архитектуры** ближе `document_sync_state_machine_proposal.md`, потому что он правильно бьёт в корень проблемы.

Но финальное решение я предлагаю строить не вокруг косметического улучшения флагов и не вокруг одного большого FSM, а вокруг следующей формулы:

**единый доменный источник истины + derive-селекторы для UI + раздельные команды `check/save/pull/resolve` + запрет прямых мутаций вне `DocumentTabsStore`.**

Для текущего проекта это самый устойчивый и современный путь:

- он уменьшает риск потери данных;
- хорошо ложится на MobX;
- не требует тяжёлой внешней библиотеки state machine;
- допускает поэтапную миграцию без резкого слома существующего UI.
