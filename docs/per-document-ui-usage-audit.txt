Аудит использования per-document stores в UI/feature-слое завершён. Код не изменялся. Проверенные файлы: - docs/document-store-manager-migration-plan.md - src/4-widgets/EventForm/EventForm.tsx - src/3-pages/Calendar/Calendar.tsx - src/5-features/EventSearch/EventSearchStore.ts - src/5-features/ProjectManager/ProjectList/ProjectList.tsx - src/5-features/ProjectManager/ProjectEditor/ProjectEditorStore.ts - src/1-app/Providers/StoreContext.ts Изменённых/созданных файлов нет. Выводы по файлам:
 1) src/4-widgets/EventForm/EventForm.tsx - Миграция на documentTabsStore.getActiveDocumentStores() выполнена. - Остался UX-риск: форма рендерится и даёт ввод даже без активного документа, а действия Save/Delete/Add/Complete молча не делают ничего из-за guard `if (!eventsStore) return`. - Критичный regression risk: форма редактирует событие из `eventFormStore`, но сохраняет/удаляет его в store текущего активного документа. Если открыть форму в документе A и переключиться на документ B до сохранения, операция пойдёт в B. - Риск устаревших данных формы: `react-hook-form` использует `defaultValues` только при инициализации, нет `reset` при смене `eventFormStore.eventData`/документа. - Архитектурная непоследовательность: `eventsCache.init()` вызывается вручную только в add-сценарии, тогда как после миграции это должно централизованно происходить через orchestration/reactivity.
 2) src/3-pages/Calendar/Calendar.tsx - Миграция на active document stores выполнена. - Drag/drop безопасно проверяет отсутствие active store, но при `!eventsStore` операция silently no-op — плохой UX без empty/disabled state. - Компонент рендерит общую модалку EventForm, что усиливает риск кросс-документного редактирования: форма может быть открыта для документа A, а активный документ уже B. - `forceUpdate()` после drag/drop выглядит как симптом неполной реактивной композиции после миграции. - Есть композиционный риск зависимости от `calendarStore/eventsCache`: даже при корректном получении active store сам Calendar останется неконсистентным, если нижележащие store/cache не полностью перестраиваются при смене активного документа.
 3) src/5-features/EventSearch/EventSearchStore.ts - Конструктор уже принимает `DocumentTabsStore`, прямая привязка к глобальному `EventsStore` удалена. - Критичный null-safety дефект: код массово использует `this.activeEventsStore!` (`planned`, `completed`, `plannedRepeatable`). При отсутствии активного документа возможен runtime crash. - Состояние поиска не привязано к documentId: при переключении вкладки сохраняются `results/currentIndex/earliestFound/latestFound`, а дальнейшие действия начнут использовать уже другой active store. Возможна логическая смесь результатов разных документов. - `toggleActive/search` допускают сценарий активного поиска без активного документа. - API по смыслу теперь scoped to active document, но это не зафиксировано явно в интерфейсе/ожиданиях.
 4) src/5-features/ProjectManager/ProjectList/ProjectList.tsx - Миграция на active `projectsStore` выполнена. - Без активного документа компонент показывает пустой список без объяснения причин. - Кнопка `Add Project` остаётся доступной даже без активного документа, хотя дальнейшее сохранение зависит от active store в `ProjectEditorStore`. - Удаление через confirm использует optional chaining и может молча не выполниться, если active document/store исчез между открытием диалога и подтверждением. - UI привязан к строковому литералу `'Default'` вместо доменной константы/флага.
 5) src/5-features/ProjectManager/ProjectEditor/ProjectEditorStore.ts - Конструктор уже принимает `DocumentTabsStore`, прямая зависимость от глобального `ProjectsStore` удалена. - Критичный null-safety дефект: в нескольких местах используется `this.activeProjectsStore!` (`getByName`, `exists`, `add`, `update`). При отсутствии активного документа или переключении/закрытии документа во время редактирования возможен runtime crash. - Критичный regression risk: редактор не закреплён за documentId. Если открыть редактирование проекта в документе A, затем переключиться на B и сохранить, операция и валидация будут происходить уже против B. - `openAdd()` позволяет открыть форму без активного документа. - `validate()` проверяет уникальность имени против текущего active store, а не store документа, где была начата операция.
 6) src/1-app/Providers/StoreContext.ts - Контекст очищен корректно: глобальные `projectsStore/eventsStore` удалены, DI идёт через `documentTabsStore`. - Но архитектурно часть feature-stores всё ещё зависит от неявного “глобального” active document, считывая его в момент действия, а не фиксируя documentId при старте операции. Это особенно опасно для долгоживущих UI-сценариев (формы, поиск, confirm dialogs). Приоритеты: - High: - src/5-features/EventSearch/EventSearchStore.ts — null-safety через `activeEventsStore!`, отсутствие reset на смену документа. - src/5-features/ProjectManager/ProjectEditor/ProjectEditorStore.ts — null-safety через `activeProjectsStore!`, запись/валидация в другом документе после переключения вкладки. - src/4-widgets/EventForm/EventForm.tsx — операции идут в store текущего активного документа, а не документа, где была открыта форма. - Medium: - src/4-widgets/EventForm/EventForm.tsx, src/5-features/ProjectManager/ProjectList/ProjectList.tsx, src/3-pages/Calendar/Calendar.tsx — отсутствие явных empty/disabled states без активного документа. - src/3-pages/Calendar/Calendar.tsx — `forceUpdate()` как smell неполной реактивности. - Low: - src/5-features/ProjectManager/ProjectList/ProjectList.tsx — литерал `'Default'`. - src/5-features/EventSearch/EventSearchStore.ts — неявная семантика поиска только по активному документу.
