## План обобщения `UnsavedChangesPrompt` и переноса в shared/ui

### 1. Цели и мотивация

- **Текущая ситуация**
  - Компонент `UnsavedChangesPrompt` реализован как фича в слое `5-features`.
  - Семантически он является диалогом подтверждения с тремя фиксированными действиями: `save` / `discard` / `cancel`.
  - Вёрстка и поведение опираются на уже существующие shared-компоненты: `Dialog`, `DialogContent`, `DialogActions`, `TextButton`.

- **Желаемое состояние**
  - Появляется **универсальный диалог подтверждения с тремя кнопками** в слое `7-shared/ui`.
  - Текст заголовка/контента и подписи кнопок настраиваются извне.
  - Колбэк `onDecision` возвращает дискретное значение, например: `'yes' | 'no' | 'cancel'`.
  - Текущий `UnsavedChangesPrompt` становится тонкой фичей-обёрткой над общим диалогом, инкапсулирующей доменный текст и маппинг решений.

- **Ожидаемые эффекты**
  - Снижение дублирования разметки трёхкнопочного диалога в разных частях приложения.
  - Централизация визуального стиля и поведения confirm-диалогов в слое `shared/ui`.
  - Повышение переиспользуемости: один и тот же базовый компонент можно использовать не только для «несохранённых изменений», но и для других сценариев.

### 2. Оценка идеи с точки зрения FSD

- **Соответствие слоям**
  - Сейчас `UnsavedChangesPrompt` живёт в `5-features`, но его содержимое почти полностью состоит из UI-композиций над shared-компонентами и жёстко захардкоженных текстов.
  - В терминах FSD:
    - **Общий трёхкнопочный диалог** — это классический `shared/ui`-компонент.
    - **Конкретный сценарий «несохранённые изменения»** — это фича, которая:
      - знает доменный текст,
      - определяет смысл вариантов (`save` / `discard` / `cancel`),
      - может добавлять доменную обвязку (например, работа с формой, навигацией и т.п.).
  - Перенос базового диалога в `shared/ui` и оставление доменной логики в `5-features` соответствует принципу: **общие абстракции вниз, доменная логика наверх**.

- **API и семантика**
  - В `shared/ui` имеет смысл использовать нейтральные значения: `yes` / `no` / `cancel`.
  - В `5-features/UnsavedChangesPrompt` можно сохранить текущий доменный тип:
    - `UnsavedDecision = 'save' | 'discard' | 'cancel'`.
    - Внутри фичи маппить `yes -> 'save'`, `no -> 'discard'`, `cancel -> 'cancel'`.
  - Такой подход позволяет:
    - не «протаскивать» доменную лексику в shared-слой,
    - сохранить обратную совместимость для текущих вызовов `UnsavedChangesPrompt`.

- **Потенциальные риски**
  - **Переусложнение API shared-компонента**: если сразу пытаться поддержать произвольное количество кнопок, разные типы иконок и т.д., можно превратить простой диалог в перегруженную абстракцию.
    - **Рекомендация:** на первом шаге ограничиться строго тремя действиями и простым API.
  - **Размывание ответственности**: важно, чтобы shared-компонент не начал знать о конкретных сценариях (unsaved, delete, reset и т.д.).
    - Для этого все тексты и семантические типы решений остаются в фичах/страницах.

- **Вывод по FSD**
  - Идея **хорошо укладывается в FSD**: отделяем доменную фичу «несохранённые изменения» от абстрактного UI-паттерна «трёхкнопочный диалог подтверждения».
  - Критично сохранить:
    - простое, нейтральное API на уровне `shared/ui`,
    - адаптационный слой в фиче для доменных типов.

### 3. Целевое API универсального диалога

Предлагаемое название: `ConfirmDialogThreeActions` (можно адаптировать под общие соглашения по неймингу в проекте).

- **Тип результата**

```ts
export type ConfirmThreeDecision = 'yes' | 'no' | 'cancel'
```

- **Пример пропсов**

```ts
type ConfirmDialogThreeActionsProps = {
  open: boolean
  title?: React.ReactNode
  description?: React.ReactNode
  yesLabel: React.ReactNode
  noLabel: React.ReactNode
  cancelLabel: React.ReactNode
  onDecision: (decision: ConfirmThreeDecision) => void
}
```

- **Пример использования для «несохранённых изменений»**

```tsx
<ConfirmDialogThreeActions
  open={open}
  title="Есть несохранённые изменения"
  description={`Что сделать перед действием "${actionName}"?`}
  yesLabel="Сохранить"
  noLabel="Не сохранять"
  cancelLabel="Отмена"
  onDecision={decision => {
    if (decision === 'yes') onUnsavedDecision('save')
    if (decision === 'no') onUnsavedDecision('discard')
    if (decision === 'cancel') onUnsavedDecision('cancel')
  }}
/>
```

### 4. План переноса и рефакторинга

#### 4.1. Создание базового компонента в `shared/ui`

- **Шаги**
  - Определить целевую директорию, например: `src/7-shared/ui/ConfirmDialogThreeActions/`.
  - Создать компонент `ConfirmDialogThreeActions.tsx`, который:
    - Использует уже существующие `Dialog`, `DialogContent`, `DialogActions`, `TextButton`.
    - Принимает пропсы согласно целевому API.
    - По клику на каждую кнопку вызывает `onDecision('yes' | 'no' | 'cancel')`.
    - При закрытии диалога через `onClose` также трактует событие как `cancel`.
  - Экспортировать компонент из index-файла слоя `shared/ui`, если такой точкой входа вы пользуетесь.

- **Особенности реализации**
  - Тексты (`title`, `description`, подписи кнопок) не хардкодятся в компоненте.
  - Компонент остаётся «глупым» и отвечает только за отображение и маршрутизацию кликов.

#### 4.2. Обновление `UnsavedChangesPrompt` в слое `5-features`

- **Шаги**
  - Оставить файл `src/5-features/UnsavedChangesPrompt/UnsavedChangesPrompt.tsx` на месте, чтобы не ломать импорт внешнего кода.
  - Обновить реализацию:
    - Заменить прямое использование `Dialog`, `DialogContent`, `DialogActions`, `TextButton` на использование `ConfirmDialogThreeActions` из `shared/ui`.
    - Сохранить публичный тип `UnsavedDecision = 'save' | 'discard' | 'cancel'`.
    - Внутри компонента маппить решение `ConfirmThreeDecision` в `UnsavedDecision`.
  - Убедиться, что публичный интерфейс фичи (`props` компонента и тип `UnsavedDecision`) остаётся прежним.

- **Преимущества**
  - Внешний код, использующий `UnsavedChangesPrompt`, не меняется.
  - Фича превращается в тонкий доменный адаптер над универсальным shared-компонентом.

#### 4.3. Поиск и адаптация всех мест использования

- **Шаги**
  - Найти все места, где используется `UnsavedChangesPrompt` (по импорту).
  - Проверить, что:
    - их логика работы с `UnsavedDecision` остаётся корректной,
    - поведение диалога визуально не изменилось (по возможности — через ручное тестирование).
  - При необходимости:
    - добавить небольшие юнит-тесты/скриншот-тесты для ключевых сценариев.

#### 4.4. Подготовка к потенциальному прямому использованию shared-компонента

- **Идея на будущее**
  - В сценариях, где не нужен доменный тип `UnsavedDecision`, страницы/фичи смогут использовать `ConfirmDialogThreeActions` напрямую.
  - Можно задокументировать типовые паттерны использования (например, для подтверждения удаления, сброса настроек и т.д.).

### 5. Технические детали и договорённости

- **Именование**
  - Если в проекте уже есть конвенции для confirm-диалогов (например, `ConfirmDialog`, `AlertDialog`), стоит:
    - проверить их,
    - либо встроить новый компонент в существующую систему, либо выбрать имя в том же стиле.

- **Типы и локализация**
  - Типы решений (`ConfirmThreeDecision`, `UnsavedDecision`) остаются на уровне TypeScript и не зависят от текста на кнопках.
  - Текст кнопок и контента может быть:
    - либо строками, захардкоженными в фичах,
    - либо полученными через систему локализации (если она есть).

- **Стили и единообразие UI**
  - Важно использовать те же кнопки (`TextButton` либо другой общий компонент), что и в остальных диалогах.
  - Если в проекте предусмотрены разные приоритеты кнопок (primary/secondary), можно позже расширить API компонента, но не делать это на первом шаге, чтобы не усложнять миграцию.

### 6. Итог

- **Идея обобщения `UnsavedChangesPrompt` до универсального трёхкнопочного диалога и переноса базовой реализации в `shared/ui` считается корректной и полезной с точки зрения FSD.**
- **Рекомендуется реализовать её в два этапа:**
  - сначала создать нейтральный `ConfirmDialogThreeActions` в `shared/ui`,
  - затем переписать фичу `UnsavedChangesPrompt` как адаптер над этим компонентом с сохранением текущего публичного API.
