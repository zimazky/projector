## Расширение синтаксиса ZCron для зависимых и отложенных событий

### 1. Цели расширения

- **Событие через интервал после другого события**  
Пример: событие B должно происходить через `d` дней/часов после события A с ID `id`.  
Пользовательский набросок: `@{id}/{d}`.
- **Повторяющееся событие после определённого события**  
Пример: после наступления события A, событие B должно запускаться многократно по своему шаблону (например, каждый день или каждую неделю), возможно с начальным сдвигом относительно A.

При этом важно:

- не сломать существующие сценарии, использующие текущий cron-подобный синтаксис;
- сохранить эффективные вычисления (не взрывать сложность при большом количестве зависимостей);
- сделать строки достаточно читаемыми и редактируемыми прямо в форме.

---

### 2. Краткое напоминание текущего синтаксиса ZCron

- **Обычный режим (cron-подобный)**:  
`days months weekdays`
  - `days`: 1–31, поддержка `*`, списков через `,`, диапазонов `a-b`, шагов `a/b`, `*/b`;
  - `months`: 1–12, аналогично;
  - `weekdays`: 0–6 (0 = Sunday), аналогично.
- **Режим “каждые N дней от начала”**:  
Строка начинается с `'/d'` → “каждые `d` дней от `startTimestamp`”, месяцы / дни недели игнорируются.

Вся логика завязана на днях (`86400` секунд), а вычисления делаются по timestamp + DateTime.

---

### 3. Требование 1: событие через интервал после другого события

#### 3.1. Вариант, близкий к вашему (`@id/d`)

Идея: использовать **префикс `@` как индикатор зависимого события**:

- `@id/d`  
где:
  - `id` — ID базового события (A),
  - `d` — интервал в днях после каждого вхождения A, когда нужно запланировать B.

**Плюсы:**

- очень компактно и визуально понятно: “@id” = “зависит от id”;
- хорошо ложится на уже существующую парадигму “шаг в днях”.

**Минусы / вопросы:**

- единицы измерения жёстко привязаны к дням — нельзя выразить “через 3 часа”;
- неочевидно, что делать, если событие A **повторяемое**:
  - ставить B только один раз после **первого** вхождения A?
  - или после **каждого** вхождения A создавать своё вхождение B?

#### 3.2. Предлагаемый расширенный вариант: `@id+offset`

Чтобы сделать выражение более гибким, предлагается использовать **сумму с оффсетом**:

- `@id+3d` — через 3 дня после события `id`;
- `@id+5d12h` — через 5 дней и 12 часов (если вы готовы доработать ZCron до часовой точности);
- `@id+0d` — в тот же день, что и событие `id` (может быть полезно для логических “цепочек” в один день).

**Формат `offset`:**

- `Nd` — дни;
- опционально можно ввести `Nh`, `Nm`, `Ns` для часов/минут/секунд, если это потребуется.

**Семантика для повторяемых A:**

- логично и удобно трактовать `@id+offset` так:  
“для **каждого вхождения** события A с ID `id` создать одно вхождение B со сдвигом `offset`”.

Таким образом:

- если A однократное — B тоже однократное, просто смещённое;
- если A повторяемое — B повторяется “параллельно” каждому появлению A.

Это поведение хорошо согласуется с идеей “зависимого события”.

---

### 4. Требование 2: повторяющееся событие после определённого события

Тут на самом деле нужно различать два класса задач:

1. **“Хвост” после одного исходного события**
  Пример: “после начала отпуска событие B должно происходить каждый день 10 дней подряд”.
2. **“Порождаемый повтор” от повторяемого события**
  Пример: “каждый раз, когда приходит зарплата (повторяемое событие A), стартует абонентская подписка (повторяемое событие B)”.

#### 4.1. Простой и практичный вариант: `@id+offset/step`

Предлагается расширить идею из пункта 3 и использовать **делитель `/step` как шаг для самого зависимого события**:

- `@id+3d/1d` — “начиная через 3 дня после каждого вхождения события `id`, каждый день”;
- `@id+7d/7d` — “каждую неделю, начиная через неделю после каждого вхождения `id`”.

**Семантика:**

- для каждого вхождения базового события A:
  - создаётся цепочка вхождений B, начинающаяся в `time(A) + offset`,
  - последующие вхождения B идут с шагом `step` (в днях).

**Плюсы:**

- полностью вписывается в уже знакомую модель `/d` в ZCron;
- для пользователя формула читается: “от события @id + offset, шагом /step”.

**Минусы / ограничения:**

- это по сути **ещё один уровень повторения**, что усложняет вычислитель (граф зависимостей событий);
- если не аккуратно реализовать, можно получить очень тяжёлые проходы по большому диапазону (особенно при вложенных зависимостях).

#### 4.2. Альтернативный вариант: `@id+offset | cron`

Более выразительный, но чуть менее компактный вариант — **разделить “точку старта” и “шаблон повтора”**:

- `@id+3d | */1 * `* — “после каждого вхождения `id` начать расписание B по cron-шаблону `*/1 * *` (каждый день)”.

Где:

- левая часть до `|` — **источник и сдвиг** (как в 3.2);
- правая часть после `|` — **обычный ZCron-шаблон**, считающийся уже от `time(A) + offset`.

**Плюсы:**

- повторно использует уже имеющийся синтаксис ZCron;
- удобно комбинировать: “запусти хвост по сложному шаблону после базового события”.

**Минусы:**

- выражение становится длиннее;
- реализация сложнее: ZCron нужно будет уметь “привязывать” cron-шаблон к смещённой базовой дате.

### 5. Эволюционное расширение vs полный пересмотр синтаксиса

#### 5.1. Эволюционный путь (рекомендуется как первый шаг)

Сохранить текущий синтаксис как “основу” и добавить **новые режимы по префиксам**:

- **Без префикса** — текущий cron-подобный синтаксис:
  - `days months weekdays`
  - `'/d'` (относительный от start).
- **Префикс `@` — зависимые события**:
  - `@id+offset` — одно зависимое событие на каждое вхождение A;
  - `@id+offset/step` — повторяющееся зависимое событие (хвост).

Внутри реализации при этом разумно:

- добавить **“компилятор” расписания**: `parse(scheduleString) → ParsedSchedule`;
- расширить структуру ParsedSchedule новыми полями:
  - `mode: 'absolute' | 'relative' | 'dependent'`;
  - `baseEventId?: number`;
  - `offset?: Duration`;
  - `step?: Duration | null`.

Это можно внедрять постепенно, сохраняя обратную совместимость с уже сохранёнными строками.

#### 5.2. Радикальный вариант: новый формат DSL / key-value

Если вы готовы в будущем “переварить” миграцию, можно рассмотреть **новый человекочитаемый DSL**, который потом сериализуется в строку:

- пример строки:
  - `mode=dependent;base=123;offset=+3d;step=1d`;
  - `mode=absolute;expr="*/4 * *"`;

**Плюсы:**

- более явные поля, меньше неочевидных перегрузок `/` и `@`;
- проще валидировать и расширять (новые ключи).

**Минусы:**

- сложнее редактировать “вручную” в маленьком текстовом поле формы;
- нужна миграция старых шаблонов и совместимость при чтении/записи.

Практичный компромисс:

- **внутри кода** перейти к нормализованной структуре (ParsedSchedule с полями `mode/base/offset/step/expr`);
- снаружи пока поддерживать “старый” и “эволюционно расширенный” текстовый формат (как описано в 5.1), постепенно двигаясь к более декларативному виду.

---

### 6. Резюме предложений

- Для **“события через интервал после другого”**:
  - базовый вариант: `@id/d` (если нужны только дни);
  - более гибкий вариант (рекомендован): `@id+offset`, где `offset` в днях/часах.
- Для **“повторяющегося события после определённого”**:
  - простой и компактный: `@id+offset/step` (шаг в днях);
  - более общий: `@id+offset | cron`, где правая часть — обычный ZCron-шаблон.
- Для **эволюции синтаксиса**:
  - сохранить текущий cron-подобный синтаксис без префикса;
  - добавить режим зависимых событий по префиксу `@` и постепенно перейти к внутреннему ParsedSchedule с явными полями;
  - рассматривать более декларативный DSL (key=value) как следующий этап, когда станет критично расширять выразительность.
